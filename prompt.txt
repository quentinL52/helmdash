# B. CHANTIER L — LANDING FINALE (agent front, PRIORITÉ 1)

## L0 — Résolution du conflit de route (préalable à tout)
- Une seule landing : conserver `src/app/page.tsx` comme source (c'est la version récente) MAIS la **déplacer dans `(marketing)/page.tsx`** (remplacement) et supprimer le doublon racine + le dossier `%28app%29`. Vérifier `npm run build` local → 0 erreur de route.

## L1 — Comparateur : Claude Cowork au lieu de ChatGPT
- Le tableau comparatif oppose désormais **Helmdash vs Claude Cowork vs Notion** (ou Cowork seul si le tableau est à 2 colonnes). Axes honnêtes et vérifiables : conçu pour le solo founder (vs généraliste) · données structurées du projet (runway, pipeline, hypothèses) vs documents/sessions · continuité (l'agent connaît ton projet en permanence) vs contexte à reconstruire · gamification/momentum · prix fixe et prévisible.
- Règle d'honnêteté : ne JAMAIS dénigrer ni prétendre que Cowork « ne peut pas » — formuler en « Helmdash est spécialisé pour X ». Chaque ligne du tableau doit rester vraie si un utilisateur de Cowork la lit. (Claims-register : ajouter la section comparateur.)

## L2 — Rubrique « Cohorte » → une seule section « Les 100 premiers »
- Supprimer le système d'onglets multiples. Une seule rubrique claire : **« Founder deal — les 100 premiers »** avec les avantages d'origine : 50 €/an verrouillé À VIE tant que l'abonnement reste actif · accès communauté privée solo founders · force de proposition et **vote sur les futures features** · badge Founding Member · accès prioritaire le 15/09.
- La phrase anti-ambiguïté obligatoire : « Rejoindre la waitlist est gratuit et n'engage à rien — les 100 premiers à souscrire au lancement obtiennent le deal. »
- **Supprimer « Parrainage »** entièrement (composant + clés i18n + toute mécanique associée). **Garder « Contact »**.

## L3 — Footer
- « © 2026 Helmdash · **Fait par un solo founder, pour les solo founders.** » (suppression de « à Lyon » — jamais validé et faux).
- **Ressources** : chaque lien cliquable et fonctionnel, sinon le lien est supprimé. Aucun lien `#` ou `localhost` toléré (ajouter au script CI claims un check `localhost|href="#"` → fail).
- **Confidentialité → `/legal/privacy` · CGU → `/legal/terms`** (pages existantes, il suffit de les câbler) · **Statut → page à créer** : v1 minimale = page statique « All systems operational » + lien vers l'uptime public (BetterStack/UptimeRobot — action humaine : créer le monitor public, 10 min) ; PAS de faux historique d'incidents.

## L4 — Bouton « Découvrir la solution » + démo mock (la grosse pièce du chantier)
À côté de « Join the waitlist » (section 2) : bouton secondaire **« Explorer la démo »** → route **`/demo`**, totalement découplée (aucune auth, aucune API produit, aucune écriture).
**Spec de la démo** :
1. **Périmètre : 4 vues maximum**, en lecture interactive : Today/Command Center (Top 3, relances dues, sweep « ✅ All tracked ») · Finances (runway chart) · Decisions (une décision avec son challenge IA affiché) · Progression (quêtes/XP). C'est la démo du produit réel, pas une visite guidée abstraite.
2. **Données** : un `DemoDataProvider` (contexte React) injecte des **fixtures réalistes** (persona : « Léa, solo founder SaaS, 8 mois de runway, 3 prospects en relance, 1 décision pricing ouverte ») dans les composants EXISTANTS — on réutilise les vrais composants, pas des copies (sinon la démo divergera du produit à chaque évolution). Les composants lisent les données via leur store : le provider hydrate le store en mode `demo=true`.
3. **Mutations neutralisées** : en mode demo, toute action d'écriture affiche un toast unique « This is a demo — join the waitlist to build yours » + CTA. Le chat agent est remplacé par 2-3 échanges pré-écrits (pas d'appel LLM — coût zéro, fiabilité totale).
4. **Bandeau permanent** en haut : « Demo mode — sample data » + bouton waitlist sticky.
5. **Mesure** : events PostHog `demo_opened`, `demo_view_changed`, `demo_action_blocked`, `demo_to_waitlist` — le taux demo→waitlist devient ta métrique de clarté produit.
6. Garde-fous : `/demo` en `noindex` (pas de SEO sur des données fictives présentées comme réelles… mais bandeau = honnête, indexable acceptable si tu préfères — reco : noindex) ; aucune donnée fictive ne doit ressembler à un témoignage (« Léa » est clairement un exemple).

## L5 — i18n & honnêteté de la landing
- Externaliser les 444 lignes vers `messages/` (namespace `landing` réécrit, purge des clés mortes), **défaut EN**, switch FR fonctionnel.
- Passer `check-landing-claims.mjs` sur la nouvelle page + ajouter les checks `localhost` / `href="#"` / termes du comparateur.
- Supprimer `cohort-badge.tsx` ou le recycler en `FounderDealBadge` branché sur `FounderDealCounter` (compteur réel ou rien).

**GATE L** : `npm run build` → 0 erreur · un seul `page.tsx` résolvant `/` · grep `localhost|href="#"|Lyon|Parrainage|ChatGPT` sur (marketing) → 0 · les 3 liens légaux ouvrent leurs pages · `/demo` complet au clavier et mobile, 0 appel réseau vers les API produit (onglet network vierge hors assets/PostHog) · Lighthouse mobile ≥ 90 · claims-register mis à jour (comparateur + demo).
